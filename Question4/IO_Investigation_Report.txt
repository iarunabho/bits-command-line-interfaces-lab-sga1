$ echo "Application started successfully" > app.log
Explanation: Redirects standard output (FD 1) into app.log instead of the screen.


$ cat app.log
Application started successfully
Explanation: Confirms the normal message was written to the log file.


$ ls /nonexistent_directory 2> error.log
Explanation: 2> redirects ONLY standard error (FD 2) into error.log, separating errors from normal output.


$ cat error.log
ls: cannot access '/nonexistent_directory': No such file or directory
Explanation: Shows the captured error message that went to the file, not the screen.


$ ls /etc /nonexistent_dir > combined.log 2>&1
Explanation: Sends stdout to combined.log, then points stderr to the same place (2>&1), capturing BOTH streams in one 
  file.

HOW LINUX MANAGES FILE I/O:
Everything is a file; processes use integer file descriptors managed by the kernel.
Every process inherits stdin(0), stdout(1), stderr(2), which can be redirected independently.
The kernel tracks open files per process (lsof, /proc/PID/fd) and enforces limits (ulimit).
Logging issues often come from redirection mistakes, stale open files, or hitting the FD limit.
